# 02.01 Frontend（实现落地：Descriptor-driven Decode，按 inst desc 配置解析 PTX 指令）

参照
- 抽象设计：`doc_design/modules/02.01_frontend_desc_driven_decode.md`
- Frontend 抽象设计：`doc_design/modules/02_frontend.md`
- Instruction System 抽象设计：`doc_design/modules/03_instruction_system.md`
- Dev Docs：`doc_dev/modules/02_frontend.md`、`doc_dev/modules/03_instruction_system.md`
- Contracts：`doc_dev/modules/00_contracts.md`

落地目标
- **必须实现**“PTX 指令按 inst desc 配置解析/约束 operand 形态”的端到端路径。
- Parser 不再硬编码 operand kind 推断；operand 的 kind/type/addr/symbol 形态由 descriptor 数据集驱动。
- decode 阶段错误必须可定位（`SourceLocation` + 可选 `inst_index`），并具备 deterministic 行为。

配置分层目标（必须支持）
- 让用户“只关心 PTX8 指令集”：通过一份 PTX ISA 映射配置把 PTX op 形态映射到通用 IR op。
- uops 组合由通用 IR 语义库提供（默认内置或由独立 JSON 提供），用户不需要编写/理解 uops。

模式约束（必须）
- PTX op →（原 inst desc 的 opcode key / `ir_op`）的映射是以后**唯一必选**模式。
- 不存在“仅靠 inst desc 就能描述 PTX 指令形态并完成解析”的模式。

落地位置（代码）
- `include/gpusim/contracts.h`
  - 原则：不新增 `InstRecordRaw` 这种对外 Contracts 类型；PTX 解析的“token 化中间态”应尽量保持为实现细节。
  - 新增/调整的重点是：让 `InstRecord` 在进入执行系统前已经是“IR op + typed operands”。
- `include/gpusim/frontend.h`
  - 可选：为 Parser 增加“只 token 化”的 API（返回内部结构或仅供 runtime 组合调用），但不强制把 raw 中间态做成公开类型。
- `src/frontend/`
  - `parser.cpp`：输出 token 化结果（带 operand_tokens，但不做 kind 推断）
  - `binder.cpp`：若绑定需要依赖 typed operands，则调整为在 decode 之后执行（见 4.2）
- `include/gpusim/instruction_desc.h` / `src/instruction/descriptor_registry.cpp`
  - 若复用 Registry 思路承载 PTX ISA map：新增候选枚举 API：`candidates()`（供“PTX→IR 映射/解析”使用）
- `src/instruction/`
  - 新增 `ptx_isa_mapper.cpp`（或沿用 `decoder.cpp` 文件名，但语义是“PTX→IR 映射/解析”）
- `src/runtime/runtime.cpp`
  - 调整调用顺序：load PTX ISA map + IR semantics → parse(tokens) → map_to_inst_records(IR) → bind（如需要）→ execute

配置文件建议（模式 B：推荐）
- PTX ISA 映射（用户提供/选择）：`ptx_isa_ptx8.json`（名称仅约定）
  - 内容：`ptx_opcode/type_mod/operand_kinds -> ir_op`
- IR 语义库（默认提供）：沿用“原 inst desc JSON”（或内置表）
  - 内容：`ir_op(inst_desc.opcode) (+operand_kinds) -> uops[]`

## 1. 类型与数据结构落地

### 1.1 中间态策略（必须）
目标是“读入 PTX 后，按 PTX ISA map 直接产出 IR 形态的 `InstRecord`”，因此不建议把 `InstRecordRaw` 变成公开契约。

推荐做法
- Parser 内部生成一个**实现细节**的 token 化结构（例如 `PtxInstTokens` / `ParsedInstLine`），字段类似：`pred/opcode/mods/operand_tokens/dbg`。
- 随后由独立的 `PtxIsaMapper`（或命名为 InstDecoder，但语义是“PTX→IR 映射/解析”）读取 token 化结构 + PTX ISA map，直接生成 `InstRecord`。

对 `InstRecord` 的约束（必须明确）
- 进入 Instruction System 之后，`InstRecord.opcode` 建议语义化为 `ir_op`（canonical op）。
- 若调试需要保留原始 `ptx_opcode/type_mod`：可在 mapper 的 Diagnostic message 中带上，或后续再考虑扩展 `InstRecord` 字段；本阶段优先不扩展 contracts。

## 2. Parser 改造（输出 raw 指令）

> 本节标题沿用“输出 raw”历史称呼，实际目标是：Parser 输出“token 化中间态（实现细节）”，而不是公开的 Raw Contracts。

目标
- 保留现有 `.entry/.param/.reg` 等解析能力。
- 指令行不再直接产出 typed operands；只解析 `pred`、`opcode`、`mods`，operands 只切分为 `operand_tokens`，不调用 `parse_operand()`。

落地点
- `src/frontend/parser.cpp`：
  - 将 `parse_operand()` 替换为 `split_operands_to_tokens()`（只做 tokenize）。
  - `parse_inst_line()` 可以改为返回内部 token 化结构（或新增 `parse_inst_line_tokens()`）。

tokenize 规则（与现有一致）
- 以 `,` 或 `;` 分割 operands。
- 对每个 token 做 trim。
- 保留方括号形式（例如 `[%rd0+4]`、`[param_name]`）作为单个 token。

## 3. DescriptorRegistry：候选枚举 API（供 decode 使用）

> 注意：在“两层配置”推荐模式下，这里更准确的对象是“PTX ISA map 的 registry”。实现上可以复用 DescriptorRegistry 的数据结构/接口思想，但语义上建议区分：
> - `PtxIsaRegistry`：`ptx_opcode/type_mod/operand_kinds -> ir_op`
> - `IrSemRegistry`：`ir_op (+operand_kinds) -> uops[]`

说明（两层配置下的角色）
- 若采用“模式 B（推荐）”：
  - decode 阶段需要对“PTX ISA 映射表”做 candidates（按 `ptx_opcode/type_mod/operand_count`）。
  - expand 阶段需要对“IR 语义库”做 lookup（按 `ir_op/(operand_kinds)`）。
- 实现上可以：
  - 方案 1：复用同一个 Registry 类型（不同数据集实例）
  - 方案 2：拆成 `PtxIsaRegistry` + `IrSemRegistry`（接口更清晰）

### 3.1 新增 API（必须）
在 `include/gpusim/instruction_desc.h` 增加：
```text
std::vector<const InstDesc*> candidates(
  const std::string& opcode,
  const std::string& type_mod,
  std::size_t operand_count
) const;
```

约束
- 返回顺序必须 deterministic。
- 排序规则：exact `type_mod` 优先于 wildcard（desc.type_mod==""）。
- 必须过滤 operand_count 不一致的 desc。
- 可选：后续可将 `mods.space/flags` 纳入过滤/排序；接口先留足扩展点。

### 3.2 Canonicalization
在 Registry 内部实现：
- opcode 统一为小写存储与比较。
- type_mod/space/flags 使用原样字符串即可，但比较需明确大小写策略（建议小写）。

## 4. InstDecoder（descriptor-driven decode）实现

> 本节的“InstDecoder”在本思路下可改名为 `PtxIsaMapper` 更贴切：输入是 token 化后的 PTX 指令行，输出直接是 IR 形态的 `InstRecord`。

### 4.1 模块位置
建议放在 `src/instruction/decoder.cpp` + `include/gpusim/` 对外暴露最小 API：
- 理由：decode 逻辑依赖 descriptor registry，且产出 `InstRecord` 供 instruction system 执行。

### 4.2 对外 API（建议）
```text
class PtxIsaMapper {
public:
  Result<InstRecord, Diagnostic> map_one(const /*tokenized*/ auto& inst, const /*ptx-isa*/ auto& reg) const;
  Result<std::vector<InstRecord>, Diagnostic> map_kernel(const std::vector</*tokenized*/ auto>& insts, const /*ptx-isa*/ auto& reg) const;
};
```

Result 形态
- 若当前代码未引入 Result 类型：可用 `std::optional<InstRecord>` + out Diagnostic，或直接抛异常并在 Runtime 捕获统一转 Diagnostic。
- 但对齐 Contracts 约束，推荐：decode 失败时返回 `Diagnostic{ module="frontend"|"instruction", code=..., location=raw.dbg }`。
 但对齐 Contracts 约束，推荐：映射/解析失败时返回 `Diagnostic{ module="frontend"|"instruction", code=..., location=inst.dbg }`。

### 4.3 解码流程（必须 deterministic）
对每条 token 化后的 PTX inst：
1) `opcode` canonicalize（lower-case）。
2) `reg.candidates(opcode, inst.mods.type_mod, inst.operand_tokens.size())`。
3) 逐 candidate 尝试 `parse_operand_by_kind(kind, token)`：
   - 全部成功则 match。
4) match==1 → 产出 IR 形态的 `InstRecord`：
  - `InstRecord.opcode = candidate.ir_op`（推荐约定）
  - `InstRecord.mods` 仍携带 type_mod/space/flags（用于后续诊断/观测或 IR semantics 的形态选择）
  - `InstRecord.operands` 为 typed operands
5) match==0 → Diagnostic：
   - 优先 `DESC_NOT_FOUND`（无候选）
   - 或 `OPERAND_FORM_MISMATCH`（有候选但均 parse 失败）
6) match>1 → Diagnostic：`DESC_AMBIGUOUS`（message 列出候选 signature）。

### 4.4 kind → token 解析（必须覆盖 schema 中所有 kind）
按 design doc 的基线规则实现以下 parser：
- `reg`：`%rN`/`%rdN`
- `pred`：`%pN`
- `imm`：十进制/十六进制（含负号）
- `addr`：`[...]` 内是 `%rdN` 或 `%rdN+imm`
- `symbol`：token 字符串（允许 `[name]` 形式作为 symbol token）

注意事项
- 解析失败必须返回 `OPERAND_PARSE_FAIL` 或被归因到 `OPERAND_FORM_MISMATCH`（二者的边界需统一：
  - token 本身语法非法：`OPERAND_PARSE_FAIL`
  - token 合法但不符合候选 kind：`OPERAND_FORM_MISMATCH`
）。

### 4.5 Binder 交互策略（需要做出选择）
当前 `Binder` 只做 `bind_first_kernel()`（选择 kernel），几乎没有 binding。
当未来 Binder 需要做 label/symbol 绑定时有两种落地路径：
- 路径 A（推荐）：Binder 分拆为 `bind_metadata(raw)` 与 `bind_operands(typed)`。
  - `bind_metadata` 在 decode 前执行（用于收集 params/reg 声明、labels）。
  - decode 产出 typed inst 后，再由 `bind_operands` 做 symbol → id/addr 等绑定。
- 路径 B：Binder 全部放到 decode 后执行，但这会要求 Binder 能处理 raw token/未绑定引用，复杂度更高。

## 5. Runtime 集成（顺序调整）

目标
- PTX→IR 映射依赖 PTX ISA map，因此 Runtime 必须在映射前加载 PTX ISA map；执行期展开依赖 IR semantics，因此也需要在 execute 前加载 IR semantics。

建议顺序
```text
ptx_isa.load(ptx_isa_map)
ir_sem.load(ir_semantics)

mod_tokens = parser.parse_ptx_file_tokens(ptx_path)
kernel_tokens = binder.bind_first_kernel(mod_tokens)     // 仅挑选/收集 metadata

insts = mapper.map_kernel(kernel_tokens.insts, ptx_isa)  // 直接生成 IR 形态 InstRecord
kernel = assemble KernelImage(name/params/reg counts/insts)
exec.run(kernel, ir_sem)
```

注意
- 若 binder 未来要绑定 operands（symbol/label），应在 decode 后增加 `binder.bind_operands(kernel, ...)`。

## 6. 诊断与观测落地

### 6.1 Diagnostic 约定
- module 推荐：
  - token 级语法错误：`frontend`
  - desc 候选/歧义/不匹配：`instruction`
- code 建议：
  - `DESC_NOT_FOUND`
  - `DESC_AMBIGUOUS`
  - `OPERAND_FORM_MISMATCH`
  - `OPERAND_PARSE_FAIL`

### 6.2 ObsControl 事件（可选但推荐）
- category=`Stream` 或 `Fetch`：
  - `decode_begin/decode_end`
  - `decode_hit/decode_miss/decode_ambiguous`

## 7. 测试计划（最小集合）

位置
- `tests/integration/`：端到端（PTX + inst_desc → trace）
- `tests/unit/`：decoder 的 token → operand 解析单元测试（如果已有单测框架）

必须覆盖用例
1) `ld.u64 (reg, symbol)`：`ld.u64 %rd0, [param_name];` decode 为 (reg, symbol)
2) `mov.u32 (reg, imm)` vs `(reg, symbol)`：`0` 与 `foo` 的消歧
3) mismatch：desc 不支持 `(reg, reg, imm)` 时，`add.u32 %r0, %r1, 1;` 报 `OPERAND_FORM_MISMATCH`
4) ambiguous：构造两个 desc 都可 parse 的 case，确保报 `DESC_AMBIGUOUS`
