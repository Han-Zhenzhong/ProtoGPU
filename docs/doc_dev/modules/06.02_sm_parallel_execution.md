# 06.02 SM 并行执行（实现落地：GlobalScheduler + SMWorker 线程池）

参照
- 抽象设计：`doc_design/modules/06.02_sm_parallel_execution.md`
- 模块图：`doc_design/arch_modules_block.diagram.puml`
- Engines：`doc_dev/modules/08_engines.md`
- SIMT Core：`doc_dev/modules/06_simt_core.md`
- Memory：`doc_dev/modules/04_memory.md`
- Observability：`doc_dev/modules/01_observability.md`

落地目标
- 支持 `sm_count = N` 的 **多 SM 并行推进**：N 个 SMWorker 宿主线程并发执行不同 CTA。
- CTA **pin 到单一 SMWorker**：CTAContext（shared/barrier/warps）在 CTA 完成前不跨线程共享。
- 在 `sm_count=1` 时保持与当前顺序实现一致的行为（作为回归基线）。
- 并行模式下：Memory/Obs 的共享部分线程安全，且有可选 deterministic 回归模式。

当前实现状态（重要）
- 代码当前是“单线程顺序执行所有 CTA/warp”：`SimtExecutor::run()` 在 `src/simt/simt.cpp` 内直接枚举 CTA→warp 并跑到结束。
- `src/runtime/engines.cpp` 目前是占位；kernel orchestration 在 `src/runtime/runtime.cpp` 中直接调用 `SimtExecutor::run()`。

## 1. 配置落地（SimConfig/AppConfig）

新增配置键（保持现有 JSON 结构：`{"sim": {...}, "observability": {...}}`）
```text
sim.sm_count: int            // 默认 1
sim.parallel: bool           // 默认 false（或 sm_count>1 自动 true，二选一）
sim.deterministic: bool      // 默认 true/false（建议默认 true 用于 CI）

sim.cta_scheduler: string    // "fifo"（预留）
sim.warp_scheduler: string   // "round_robin"（预留）

observability.ts_mode: string // "tick" | "logical_clock"（默认 tick；并行建议 logical_clock）
```

落地位置（代码）
- 配置结构：`include/gpusim/simt.h` 中的 `SimConfig` 扩展字段。
- 配置解析：`src/runtime/runtime.cpp` 的 `load_app_config_json_text()` 增加字段读取（保持向后兼容：字段缺失用默认值）。

约束
- `deterministic=true` 基线策略：强制 `parallel=false` 或强制 `sm_count=1`（避免 golden traces 不稳定）。

## 2. 新增组件与代码落点

> 目标是最小侵入：先在 Runtime 侧引入 GlobalScheduler/SMWorker，并把现有 `SimtExecutor` 的执行改为可由 SMWorker 调用的“CTA/warp 推进”。

### 2.1 GlobalScheduler（CTA Distributor）
职责
- 维护待执行 CTA work items 队列，供多个 SMWorker 并发获取。
- 维护 CTA 完成计数，供 ComputeEngine/Runtime 判定 kernel complete。

建议落地位置
- `src/runtime/global_scheduler.{h,cpp}`（Runtime/ComputeEngine 侧组件）

建议接口（实现级）
```text
struct CtaWorkItem { kernel_id, cta_id, entry, launch, param_handle ... }

class GlobalScheduler {
  submit(CtaWorkItem)
  try_acquire(SmId sm_id) -> optional<CtaWorkItem>
  mark_done(CTAId)
}
```

实现要求
- 线程安全：内部用 `std::mutex + std::condition_variable`（或无锁队列，后续优化）。
- `try_acquire` 支持阻塞等待与 stop 信号（避免 busy spin）。

### 2.2 SMWorker 线程池
职责
- 启动 `sm_count` 个 worker 线程。
- 每个 worker 独占一个 `SmState`（WarpScheduler/Executor/Units）并循环处理 CTA。

建议落地位置
- `src/simt/sm_worker.{h,cpp}`：SMWorker、SmState、CTAContext 初始化逻辑（与 SIMT Core 耦合）。
- `src/runtime/sm_worker_pool.{h,cpp}`：线程池生命周期管理（与 Runtime 耦合）。

线程模型（实现级约束）
- SMWorker 线程：只推进 compute（warp steps）。
- Runtime/主线程：负责 I/O（workload/CLI）、提交 CTA work items、等待完成、收集 trace/stats。

停止协议
- pool 析构或 runtime 结束时发 stop；worker 从 condvar 醒来退出。

## 3. CTAContext/Shared/Barrier 的实现边界

关键不变量
- **CTAContext 不跨线程共享**：`shared memory` 与 `barrier state` 放在 CTAContext 内，由获得该 CTA 的 SMWorker 独占直到 CTA 完成。

落地要求
- 将现有“在 `SimtExecutor::run()` 内创建 `WarpState` 并跑完”的逻辑拆为：
  - `init_cta_context(item) -> CTAContext`
  - `warp_scheduler.pick_ready_warp(cta_ctx)`
  - `executor.step_warp(sm_state, cta_ctx, warp_id)`

备注
- 目前 `AddrSpaceManager` 只实现了 global/param；要支持 CTA 私有 shared，需要在 Memory 层补齐 shared/local/const/param 的抽象或在 CTAContext 内挂一个 shared byte-array，并让 `MemUnit` 的 shared 路由能访问到它。

## 4. Memory 的线程安全（并行必需）

现状
- `AddrSpaceManager`（`include/gpusim/memory.h` / `src/memory/memory.cpp`）无锁，不可并发访问。

基线落地（先正确）
- 在 `AddrSpaceManager` 内部增加 `std::mutex`：保护 `alloc_global/write_global/read_global/set_param_*`。
- 原子/栅栏语义基线采用“按地址或全局序列化”（先满足功能正确与可复盘）。

优化（后续）
- 分段锁（按页/区间）提升吞吐。

---

## 5. Observability 与时间戳

现状
- `ObsControl/TraceBuffer/Counters` 已用 `std::mutex`，可并发调用（见 `include/gpusim/observability.h`）。
- 但 `SimtExecutor::run()` 当前用局部 `ts++`，并行后会失效。

落地策略
- 新增全局逻辑时钟：`std::atomic<std::uint64_t> logical_clock`（建议放在 Runtime 或 ObsControl 外围）。
- 并行模式下：所有 worker 事件 `e.ts = logical_clock.fetch_add(1)`。
- deterministic 基线：禁用并行，让 tick 驱动单调 ts（与 `doc_dev/modules/08_engines.md` 一致）。

## 6. 迁移步骤（按最小风险拆分）

Step 0：只写骨架，不引入并行
- 引入 GlobalScheduler/SMWorkerPool，但 `sm_count=1` 且 worker 在同线程执行（或单 worker 线程）。
- 目标：执行路径从“直接 `SimtExecutor::run()`”迁移到“CTA work item + worker loop”，但结果一致。

Step 1：实现 CTA 内多 warp 交织推进（单线程）
- 将 CTA 的 warps 存入 CTAContext。
- WarpScheduler 每次挑一个 ready warp 调 `step_warp()`。
- 目标：为 barrier/mem blocked 留出调度空间（不要求周期精确）。

Step 2：开启多 SMWorkers 并行
- `sm_count>1 && parallel=true` 时启动多个 worker。
- 首先用 Memory 全局锁保证正确。

Step 3：确定性回归模式
- `deterministic=true`：强制 `sm_count=1` 或 `parallel=false`。

## 7. 测试与验收（dev 侧）

最小回归
- `sm_count=1`：运行 demo workload，trace/stats 与原路径一致。

并行 smoke
- `sm_count=2`：确保 trace 中出现不同 `sm_id`（需要在事件中填充 `sm_id` 字段）。
- kernel 不含跨 CTA 通信时：输出 buffer 与 `sm_count=1` 一致。

并发安全
- 压力用例：多 CTA + 大量 global 写；不崩溃、不出现 data corruption。
