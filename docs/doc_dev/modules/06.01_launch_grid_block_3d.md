# 06.01 Kernel Launch（实现落地：3D grid/block + builtins）

参照
- 抽象设计：`doc_design/modules/06.01_launch_grid_block_3d.md`
- SIMT Core 抽象设计：`doc_design/modules/06_simt_core.md`
- Runtime/Streaming 抽象设计：`doc_design/modules/07_runtime_streaming.md`
- Contracts：`doc_dev/modules/00_contracts.md`
- SIMT Core：`doc_dev/modules/06_simt_core.md`

落地目标
- 支持一次 kernel launch 的 3D `grid_dim` / `block_dim`（`x/y/z`），并将其落地为 CTA/Warp/Lane 的确定性枚举。
- 支持 PTX builtin（基线）：`%tid.{x,y,z}`、`%ntid.{x,y,z}`、`%ctaid.{x,y,z}`、`%nctaid.{x,y,z}`。
- builtin 暴露方式采用抽象设计 06.01 的“选项 A：`OperandKind::Special`”。
- 维度非法、builtin 名称非法时输出稳定 `Diagnostic`，并可定位到 launch 或 inst_index。

落地位置（代码）
- Contracts：`include/gpusim/contracts.h` + `src/common/contracts.cpp`（若需要辅助函数）
- Frontend tokenization：`include/gpusim/ptx_tokens.h`、`src/frontend/parser.cpp`
- PTX ISA mapping：`include/gpusim/ptx_isa.h`、`src/instruction/ptx_isa.cpp`、`schemas/ptx_isa.schema.json`
- SIMT/Units：`include/gpusim/units.h`、`src/simt/simt.cpp`、`src/units/exec_core.cpp`、`src/units/mem_unit.cpp`
- Runtime/CLI：`include/gpusim/runtime.h`、`src/runtime/runtime.cpp`、`src/apps/cli/main.cpp`

## 1) Contracts：Dim3/LaunchConfig/Builtins + OperandKind::Special

### 1.1 新增基础类型
在 `include/gpusim/contracts.h` 落地：

- `Dim3 { u32 x, y, z }`
- `LaunchConfig { Dim3 grid_dim, block_dim; u32 warp_size }`
- `Builtins { Dim3 tid, ntid, ctaid, nctaid; u32 laneid; u32 warpid }`

约束（必须在运行时校验）
- `grid_dim.{x,y,z} >= 1`，`block_dim.{x,y,z} >= 1`。
- 计算 `threads_per_block = block_dim.x * block_dim.y * block_dim.z` 时必须检查乘法溢出。

诊断
- 维度为 0：`runtime:E_LAUNCH_DIM`
- 乘法溢出或 `threads_per_block==0`：`runtime:E_LAUNCH_OVERFLOW`

### 1.2 builtin 暴露：OperandKind::Special
扩展 `OperandKind`：
- `OperandKind += Special`

扩展 `Operand`：
- 增加字段 `std::string special;`（保存规范化后的 builtin 名称，如 `"tid.x"`）。

命名规范（基线）
- 必须使用小写：`tid.x`/`tid.y`/`tid.z`、`ntid.x`...、`ctaid.x`...、`nctaid.x`...
- Frontend 解析 `%tid.x` 时去掉前缀 `%`，并输出规范名 `tid.x`。

实现建议
- 为避免 string 比较散落各处，提供集中函数：
  - `bool is_special_name(const std::string&)`
  - `std::optional<std::uint32_t> read_special_u32(const std::string& name, const Builtins& b)`

错误处理
- 执行阶段遇到未知 `special`：`units.exec:E_SPECIAL_UNKNOWN`（携带 `inst_index`）。

## 2) Frontend：tokenization 识别 builtin 为 Special

目标
- tokenization 仅做“识别 + 规范化”，不做任何语义求值。

落点
- `src/frontend/parser.cpp`：在 operand 解析处增加 builtin 识别分支。

识别规则（基线）
- 形如 `%tid.x` / `%ctaid.y` 的 token：
  - 若去掉 `%` 后命中允许集合，则输出 `OperandKind::Special` 且 `operand.special=<normalized>`。
  - 否则保持原有行为（通常会被当作 `Reg/Pred/Symbol` 等；若以 `%` 开头但不合法，可直接报 `frontend:PTX_PARSE_SPECIAL_UNKNOWN`）。

建议的诊断（可选）
- `%foo.bar` 这种“像 builtin 但不认识”的输入：`frontend:PTX_PARSE_SPECIAL_UNKNOWN`，location 指向该 operand。

## 3) PTX ISA map：支持 operand_kinds=Special，并能产出 InstRecord

### 3.1 Schema 与资产
- `schemas/ptx_isa.schema.json`：在 `operand_kinds[]` 允许值里加入 `"special"`。
- demo ISA map（可选）：为包含 builtin 的 PTX 指令添加映射条目（例如 `mov.u32 (reg, special) -> mov`）。

### 3.2 Mapper：operand 解析与候选匹配
在 `src/instruction/ptx_isa.cpp`：
- `operand_kind_from_string()`/等价逻辑：加入 `special`。
- operand try-parse：加入 `Special` 分支（从 token 取 `special` 字段，并写入 `InstRecord::operands`）。

不变量
- ISA mapping 的“匹配决策”仍仅依赖 `(ptx_opcode, type_mod, operand_kinds)`。
- `Special` 只是一种 operand kind，不影响排序/消歧规则。

## 4) SIMT Core：3D 枚举 + builtin 计算（增量落地）

说明
- 当前实现是单 warp 的功能执行器；要让 builtin 生效（例如 `mov %r0, %tid.x` 在不同 lane 得到不同值），必须具备 lane-wise 的寄存器/谓词语义。
- 本文按“最小可用增量”描述落地步骤：先把 Units/warp state 升级为 lane-wise，再落地 launch 3D 枚举。

### 4.1 WarpState：升级为 lane-wise（前置依赖）
在 `include/gpusim/units.h` 的 `WarpState`：
- 将寄存器与谓词改为 lane-wise 存储（推荐：`reg_major`，即 `[reg][lane]` 的扁平数组）。

建议布局（示意）
- `r_u32`: `std::vector<std::uint32_t>`, size=`reg_u32_count * warp_size`
- `r_u64`: `std::vector<std::uint64_t>`, size=`reg_u64_count * warp_size`
- `p`: `std::vector<std::uint8_t>` 或 `std::vector<bool>`，size=`pred_count * warp_size`

并提供访问器（必须）
- `read_reg(reg_id, lane)` / `write_reg(reg_id, lane, v)`

### 4.2 Builtins：每 lane 可访问
在 `WarpState` 增加 builtin 视图：
- `LaunchConfig launch`（或引用/指针）
- `Dim3 ctaid` + `std::uint32_t warpid`（warp 在 CTA 内序号）
- `std::uint32_t lane_base_thread_linear`（该 warp 的第一个 lane 对应的 thread_linear）

计算规则（与抽象设计一致）
- `thread_linear = lane_base_thread_linear + laneid`
- 由 `thread_linear` 反推 `tid.{x,y,z}`（使用 `block_dim` 做反解）：
  - `tid.x = thread_linear % block_dim.x`
  - `tid.y = (thread_linear / block_dim.x) % block_dim.y`
  - `tid.z = thread_linear / (block_dim.x * block_dim.y)`

注意
- 对部分 warp：lane 对应的 `thread_linear` 若 `>= threads_per_block`，该 lane 必须在 `active_mask` 中关闭，且不应读写寄存器/内存。

### 4.3 ExecCore：读取 Special
在 `src/units/exec_core.cpp`：
- 将 `read_imm_or_reg()` 扩展为 `read_operand_lane(op, warp, lane)`：
  - `Imm`：直接返回
  - `Reg`：按 lane 读寄存器
  - `Special`：从 builtin 计算结果读（返回 u32，必要时零扩展到 u64）

错误处理
- `Special` 名称无法解析：返回 `StepResult.diag=units.exec:E_SPECIAL_UNKNOWN` 并 `blocked_reason=Error`。

### 4.4 MemUnit：按 lane 求值地址/数据
在 `src/units/mem_unit.cpp`：
- 地址与数据读取都应迁移到 `read_operand_lane()`，确保未来 `addr`/`reg` 都是 lane-wise。
- `Symbol`（param）读取保持现有行为。

### 4.5 SimtExecutor：实现 3D grid/block 的 CTA/warp 枚举
在 `src/simt/simt.cpp`：
- 修改 `SimtExecutor::run()` 接口使其接收 `LaunchConfig`（建议：`run(kernel, launch)`）。
- 外层枚举 CTA：`ctaid.z/y/x`。
- CTA 内枚举 warp：
  - `threads_per_block = block_dim.x * block_dim.y * block_dim.z`
  - `warp_count = ceil(threads_per_block / warp_size)`
  - `warpid in [0, warp_count)`
  - `lane_base_thread_linear = warpid * warp_size`
  - `active_mask` 仅打开 `thread_linear < threads_per_block` 的 lanes

执行边界（与 06 SIMT Core 对齐）
- `warp.pc` 从 0 开始。
- 每个 warp 独立执行到 `ret` 或步数上限。
- 观测事件建议逐步补齐 `cta_id/warp_id/thread_id`（至少先补齐 `cta_id/warp_id`）。

## 5) Runtime/CLI：传入 grid/block，并与 Streaming payload 对齐

说明
- Streaming 模式下 `KernelPayload { grid_dim, block_dim }` 是唯一来源。
- 非 streaming 的 demo/CLI 可先用 flags 或 config 提供默认值。

落点（建议）
- CLI：增加 `--grid x,y,z`、`--block x,y,z`（默认 `1,1,1`）。
- Runtime：`run_ptx_kernel*()` 增加可选参数 `LaunchConfig` 或 `Dim3 grid, Dim3 block`，并在调用 SIMT 前完成校验。

诊断
- 维度非法在进入 SIMT 前报 `runtime:E_LAUNCH_DIM`/`runtime:E_LAUNCH_OVERFLOW`。

## 6) 验收与测试（建议）

### 6.1 单元测试
- Frontend：`%tid.x` 解析为 `OperandKind::Special` 且 `special=="tid.x"`。
- PTX ISA mapper：`mov.u32 (reg, special)` 能映射到 IR `mov`，且 InstRecord 中 operand kind 为 `Special`。
- Units：对固定 `LaunchConfig + ctaid + warpid`，`read_special_u32("tid.x", lane)` 在 lane 0..N 递增。

### 6.2 集成测试（golden）
- PTX kernel：把 `%tid.x`（和/或 `%ctaid.x`）写入 global，host 回读验证。
- 覆盖用例：
  - `block=(32,1,1)`：满 warp
  - `block=(40,1,1)`：部分 warp mask
  - `block=(4,3,2)`：3D 反解 tid

## 7) 兼容性说明（Option B 的保留）

若短期不希望扩展 contracts：
- 可按抽象设计 06.01 的 Option B，在每个 CTA/warp 启动时执行 prolog 将 builtin 写入约定寄存器。
- 该方案需要额外定义“约定寄存器命名/编号”，会侵入 PTX ISA map 与测试资产；不建议作为基线。
