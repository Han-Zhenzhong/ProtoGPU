# 07.01 Stream 输入（实现落地：WorkloadSpec → Commands/Streams/Deps）

参照
- 抽象设计：`doc_design/modules/07.01_stream_input_workload_spec.md`
- Runtime/Streaming：`doc_dev/modules/07_runtime_streaming.md`
- Kernel I/O + ABI：`doc_dev/modules/09_kernel_io_and_abi.md`
- Kernel Launch（grid/block + builtins）：`doc_dev/modules/06.01_launch_grid_block_3d.md`
- Contracts：`doc_dev/modules/00_contracts.md`

落地目标
- 提供一个可重放的用户输入：`WorkloadSpec(JSON)`，表达 buffers/modules/streams/commands。
- WorkloadSpec 能 deterministically 生成：`Command` 序列（带稳定 `cmd_id/stream_id/event_id`），并交给 Runtime/Streaming 调度。
- 失败时输出稳定 `Diagnostic`：schema 错、引用错误、越界/未分配、launch 维度非法等。

落地位置（代码建议）
- `src/runtime/`
  - `workload_loader.{h,cpp}`：读取 JSON + schema 校验 + 结构化解析
  - `workload_runtime.{h,cpp}`：把 WorkloadSpec 物化为 resources + commands 并执行（tick 到 sync）
  - 复用现有 `runtime.cpp` 的 HostBuf/DevicePtr/KernelArgs 等能力
- `schemas/`
  - `workload.schema.json`：WorkloadSpec schema（新增）
- `src/apps/cli/main.cpp`
  - 新增 `--workload <json>`：优先走 workload 模式

## 1) CLI 入口与运行模式

新增参数
- `--workload <path.json>`：运行 WorkloadSpec。

行为约定
- 若提供 `--workload`：忽略（或禁止组合）当前单-kernel 直跑参数（例如 `--ptx/--ptx-isa/--inst-desc`）。
- 允许保留单-kernel模式作为最小 demo；workload 模式是完整 streaming 验收入口。

输出
- trace/stats 输出与现有 CLI 保持一致（只是在 trace 中会有 STREAM/COPY/KERNEL 的 cmd 事件）。

## 2) WorkloadSpec 结构落地（JSON → 内部模型）

内部结构建议（runtime 内部，不要求对外稳定）
```text
Workload {
  host_bufs:   map<string, HostBufId>
  device_bufs: map<string, DevicePtr>
  modules:     map<string, ModuleDeclResolved>
  streams:     map<string, StreamId>
  events:      map<string, EventId>
}

ModuleDeclResolved {
  ptx_path, ptx_isa_path, inst_desc_path
  // 可选：预解析后的 kernel_tokens / kernel_image cache
}
```

ID 分配规则（必须 deterministic）
- `HostBufId/StreamId/EventId/CmdId` 由 runtime 递增分配。
- 为保证“同一份 workload 在不同机器上复现同样 cmd 因果链”，建议：
  - 对 `buffers/modules/streams` 的 map 按 key 排序后依次分配 ID。
  - 对 `commands` 按数组顺序分配 `CmdId`。

## 3) Schema 校验与错误分类

### 3.1 Schema 文件
新增
- `schemas/workload.schema.json`

校验策略（基线）
- CLI/Runtime 读取 JSON 后先做 schema 校验，再进入语义解析。

诊断
- schema 错误：`runtime:E_WORKLOAD_SCHEMA`
  - message：包含“哪一个字段缺失/类型不对/oneof 不匹配”的摘要
  - location：可选（若 JSON parser 支持位置信息）

### 3.2 语义引用校验
引用错误统一为
- `runtime:E_WORKLOAD_REF`

典型触发
- copy 引用不存在的 buffer 名称
- kernel 引用不存在的 module/stream
- event_wait 引用不存在的 event 名称

## 4) Resources 物化（buffers/modules/streams/events）

### 4.1 Host buffers
落地
- 用 `Runtime::host_alloc/host_write` 分配与初始化。

init 处理（基线）
- `zeros`：默认全 0
- `hex`：解析十六进制字符串写入（越界报错）
- `file`：读取文件 bytes 写入（越界报错）

诊断
- 越界：`runtime:E_HOST_OOB`（若 09 模块已定义）或归类为 `runtime:E_WORKLOAD_REF`（不推荐）

### 4.2 Device buffers
落地
- 用 `Runtime::device_malloc(bytes, align)` 分配。

诊断
- bytes==0 / align 非法：`runtime:E_DEVICE_ALLOC`

### 4.3 Modules
落地
- `ModuleDecl { ptx, ptx_isa, inst_desc }` 只是路径绑定。
- 执行 kernel command 时再加载（lazy）或提前加载缓存（eager）。

建议
- 基线：lazy 加载（每个 module 首次使用时 parse+map+load desc）。
- 后续可加 cache：避免同 module 多次 launch 重复 parse。

### 4.4 Streams / Events
落地
- streams：为每个 `streams` 条目分配 `StreamId` 并创建 `StreamQueue`。
- events：从 workload 的 string name 建立 `EventId` 映射；实际 signaled 状态在 DependencyTracker。

## 5) Commands 生成（WorkloadSpec → Command）

### 5.1 Copy
CopySpec → `Command{kind=COPY, payload=CopyPayload}`
- H2D：`src=host_buf(name, offset?)` → `dst=device_ptr(name, offset?)`
- D2H：`src=device_ptr(...)` → `dst=host_buf(...)`
- D2D：device→device
- MEMSET：dst device + value/bytes

诊断
- 引用不存在：`runtime:E_WORKLOAD_REF`
- bytes 越界/未分配：由 memory/runtime 对应模块报 `memory:E_*` 或 `runtime:E_*`（需带 `cmd_id/stream_id`）

### 5.2 Kernel launch
KernelLaunchSpec → `Command{kind=KERNEL, payload=KernelPayload}`

需要绑定的三类输入
- module/entry：决定 PTX 与 inst desc
- launch 维度：`grid_dim/block_dim`（对齐 06.01）
- args：对齐 09 模块的 KernelArgs 打包

诊断
- entry 不存在：`runtime:E_ENTRY_NOT_FOUND`
- launch 维度非法：`runtime:E_LAUNCH_DIM` / `runtime:E_LAUNCH_OVERFLOW`
- args 缺失或类型不匹配：`runtime:E_BAD_ARGS`

### 5.3 event_record / event_wait
- `event_record(name)`：生成 `Command{kind=EVENT_RECORD, event_id=...}`
- `event_wait(name)`：生成 `Command{kind=EVENT_WAIT, event_id=...}`

依赖语义
- 完全复用 07 模块：由 DependencyTracker 维护 signaled 状态与 ready 判定。

### 5.4 sync
语义
- `sync {}` 是“显式同步点”：要求该 stream 的队列推进到 sync 之前的命令全部 complete 后才算 sync 完成。

落地选项
- 选项 A（推荐基线）：把 `sync` 也落成一个 `Command{kind=SYNC}`（需要扩展 CommandKind），并在 ready/complete 逻辑中把它当作 barrier。
- 选项 B（不改 CommandKind）：CLI 看到 sync 时，执行一个 `drain_stream(stream)` 过程（不进入队列）。

推荐
- 选项 A：可观测、可复盘（trace 可看到 sync 作为 cmd 节点）。

## 6) 调度执行与可观测（对齐 07_runtime_streaming）

执行循环（基线）
- enqueue：把每个 stream 的 commands 依次入队（产生 `STREAM:cmd_enq`）
- tick：按 07 模块，依次 peek 队首，is_ready 则 submit 到 engine
- complete：engine 完成回调 dep tracker，产生 `STREAM:cmd_complete`

建议事件（至少）
- `STREAM:cmd_enq/cmd_ready/cmd_submit/cmd_complete`
- `STREAM:event_record/event_wait_done`
- `COPY:*`（copy engine）与 `KERNEL:*`（compute engine）可按现有实现逐步补齐

事件字段要求
- 必须携带：`cmd_id/stream_id`；event 相关必须携带 `event_id`。

## 7) 验收用例（集成测试入口）

最小回归（单 stream）
- host 输入 H2D → kernel(grid=1, block=32) → D2H → sync
- host 断言输出 bytes 与预期一致

跨 stream 依赖
- stream0: recordEvent(E)
- stream1: waitEvent(E) → kernel
- 断言：kernel 的 cmd_submit 必须在 E signaled 后

3D launch
- grid=(2,2,1)、block=(16,2,1)
- kernel 写回 `ctaid/tid`（对齐 06.01）供 host 回读验证
