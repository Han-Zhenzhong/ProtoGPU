# 01 Fail-fast：global OOB/未分配访问必须报错

本文件是 [docs/doc_spec/sm70_profile.md](../../doc_spec/sm70_profile.md) 附录 B（Memory baseline）在实现层的落地说明，目标是防止未来回归把 OOB 写重新变成“静默成功”。

---

## 1) 冻结口径（必须满足）

- 对未分配/越界的 global 访问：必须 fail-fast 并返回 `Diagnostic`。
- 禁止：
  - `ld.global` 缺页时返回 0
  - `st.global` 越界时静默忽略或静默写入

---

## 2) 当前实现落点

### A. 分配边界（allocation ranges）

- 全局内存分配：`Runtime::device_malloc()` → `AddrSpaceManager::alloc_global()`
- 地址范围记录：`AddrSpaceManager::allocs_`

文件
- `include/gpusim/memory.h`
- `src/memory/memory.cpp`

### B. 读写检查

- `AddrSpaceManager::read_global(addr,size)`：
  - 若访问范围不落在任一 allocation 内，返回 `std::nullopt`。

- `AddrSpaceManager::write_global(addr,bytes)`：
  - 返回 `bool`：范围合法则写入并返回 true，否则返回 false。

这保证 OOB 能在最底层被识别出来。

### C. 执行期诊断

- `src/units/mem_unit.cpp`
  - `ld.global.*`：对齐检查 + `read_global` 失败返回 `E_GLOBAL_MISS`
  - `st.global.*`：对齐检查 + `write_global` 失败返回 `E_GLOBAL_MISS`

---

## 3) 回归用例（必须存在）

- fixture：`tests/fixtures/ptx/mem_oob_st_global_f32.ptx`
  - 策略：先 `ld.global.f32 [%out]`（确保有效读），再 `st.global.f32 [%out+4]`（对 4-byte allocation 越界写）
  - 注意：用 `[%rd+imm]` 形式避免依赖 `add.u64 (reg,reg,imm)` 的 form

- 测试入口：`tests/tiny_gpt2_minimal_coverage_tests.cpp`
  - `test_oob_st_global_must_fail_fast()` 断言：
    - `completed == false`
    - `diag.has_value() == true`
    - 该 fixture 设计为“对齐但越界”，因此诊断码应为 `E_GLOBAL_MISS`

- 额外回归（对齐检查单独锁死）：
  - fixture：`tests/fixtures/ptx/mem_misaligned_ld_global_f32.ptx`
  - 断言：`diag.code == E_GLOBAL_ALIGN`

---

## 4) 常见坑

- fixture 中用 `add.u64 %rd2, %rd1, 4` 计算地址：如果 `add.u64 (reg,reg,imm)` 未被纳入 bring-up forms，会导致测试提前在 mapper/descriptor 阶段失败，覆盖不到真正的 OOB 行为。
- 只在 `MemUnit` 里检查 OOB，但底层内存模型允许写入新地址：会导致未来其它路径（例如 memcpy）绕过检查。

建议
- 同时在内存模型和 MemUnit 两层做防御：
  - 内存模型保证“不会写出 allocation 范围”
  - MemUnit 保证“失败会变成可定位 Diagnostic”
