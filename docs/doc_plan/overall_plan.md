# Overall Plan（总体路线）

本文给出本工程从“抽象逻辑设计”到“测试实现”的总体路线与产出物归档位置。

全局约束
- 设计基准以 PUML 图为准（尤其是 doc_design/arch_modules_block.diagram.puml 与 doc_design/sequence.diagram.puml）。
- 指令描述格式固定为 JSON。

---

## Phase 1：抽象逻辑设计（Architecture / Logical Design）

目标
- 完成系统架构、模块边界、端到端流程、跨模块契约（Contracts）。

主要产出（归档位置）
- doc_design/：架构设计文档与 PUML 图（模块图、时序图等）。
- doc_plan/plan_design/：模块设计的分步计划、设计路线与设计输出清单。

完成标准
- 所有模块在 PUML 图中可定位；模块职责、输入/输出、交互接口契约与关键状态机均在设计文档中有明确描述。

---

## Phase 2：代码开发设计（Implementation Design for Development）

目标
- 将抽象逻辑设计落为“指导代码开发”的实现级设计：代码结构、模块落地方案、关键类/文件拆分、错误处理与日志/观测落地方式、配置与构建策略。

主要产出（归档位置）
- doc_dev/：开发实现设计（代码组织、接口落地、规范、开发流程）。
- doc_plan/plan_dev/：开发阶段的任务拆解与验收点计划。

完成标准
- 每个逻辑模块在代码中有明确落地边界与依赖关系；实现级接口与数据结构可直接指导编码。

---

## Phase 3：代码开发（Implementation）

目标
- 按 doc_dev 的实现设计完成代码实现与集成，保证与 PUML 流程与契约一致。

主要产出
- 源码与可运行产物（具体路径随后续项目结构确定）。

完成标准
- 核心功能按架构流程可运行；关键路径有观测输出可复盘（Trace/Counters）。

---

## Phase 4：测试设计（Test Design）

目标
- 定义测试策略与覆盖面：功能正确性、SIMT 语义、stream/event 依赖、copy/compute、内存语义（atomic/fence/barrier）与可观测一致性。

主要产出（归档位置）
- doc_tests/：测试策略、用例清单、验收场景、golden traces 约定与回归规则。
- doc_plan/plan_test/：测试阶段的计划与用例拆解。

完成标准
- 测试范围、用例分层、数据准备方式、判定标准与回归策略完整明确。

---

## Phase 5：测试代码实现设计（Implementation Design for Tests）

目标
- 将测试设计落为可编码的测试实现方案：测试框架选型、目录结构、测试工具链、golden trace 格式与生成/对比机制、Mock/Fixture 方案。

主要产出（归档位置）
- doc_dev/ 或 doc_tests/（按需要选择其一作为“测试实现设计”的承载处）：
  - 推荐：测试实现细节写在 doc_tests/，与测试策略同处；与工程开发工具链强相关的部分写在 doc_dev/。

完成标准
- 测试代码可以直接按设计落地，且与被测模块边界与观测事件契约一致。

---

## Phase 6：测试代码实现（Test Implementation）

目标
- 完成测试代码实现与持续回归执行，形成可重复的验证闭环。

主要产出
- 测试代码、测试数据、golden traces、CI/执行脚本（具体路径随项目结构确定）。

完成标准
- 测试可自动运行；覆盖关键验收场景；失败可定位并可复现。

---

## Doc Plan A：Build Doc（doc_build/）书写计划

目标
- 让工程在任意干净环境中具备可重复的“获取依赖 → 构建 → 生成文档/产物 → 运行基本验证”的说明与脚本约定。

主要产出（归档位置）
- doc_build/：
  - 文档构建入口与约定（例如：文档生成命令、输入/输出目录、依赖工具版本要求）。
  - CI/本地一致的 build 流程说明（如何跑、如何排错、常见问题）。

写作时机
- 在 Phase 6（测试代码实现）完成并完成第一版交付后，再开始撰写与整理。

完成标准
- doc_build/ 中存在唯一主入口文档，能从零说明：依赖要求、构建命令、文档生成命令、产物位置、测试/验证入口。

---

## Doc Plan B：User Doc（doc_user/）书写计划

目标
- 面向使用者说明“这个项目是什么、如何运行、如何配置、如何查看输出/trace、如何定位问题”。

主要产出（归档位置）
- doc_user/：
  - 快速开始（Quick Start）：从准备环境到跑通一次示例。
  - 使用指南：运行方式、配置文件说明、输入文件（如 PTX/descriptor JSON）与输出（trace/stats）说明。
  - 常见问题与排错：常见错误与对应定位路径。

写作时机
- 在 Phase 6（测试代码实现）完成并完成第一版交付后，再开始撰写与整理。

完成标准
- doc_user/ 中存在唯一主入口文档，能在不阅读设计文档的情况下完成：安装/运行/配置/查看输出/常见问题排查。
