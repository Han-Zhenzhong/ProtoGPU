# 06.02 SM 并行执行（每 SM 一个软件线程）

说明
- 本文描述 **多 core（SM）并行仿真** 的抽象逻辑：CTA 分发、SM worker 线程模型、与 SIMT/Memory/Observability 的交互。
- 设计基准：与 [doc_design/arch_modules_block.diagram.puml](../arch_modules_block.diagram.puml) 与 [doc_design/sequence.diagram.puml](../sequence.diagram.puml) 对齐；若文本与图冲突，以图为准。
- SIMT Core 的单步语义与上下文模型详见：[doc_design/modules/06_simt_core.md](06_simt_core.md)
- Engines 的提交/tick/完成信号详见：[doc_design/modules/08_engines.md](08_engines.md)

---

## 目标（Goals）
- 支持 **多 SM 并行执行**：`sm_count = N` 时，启动 N 个 SM worker，每个 worker 代表一个“core/SM”。
- 支持 **CTA 粒度分发**：Global Scheduler 从待执行 CTA 队列选择下一个 CTA，并分配给空闲 SM。
- 保持 **No-cache 内存模型**：不仿真 L1/L2/NoC/DRAM 微结构；只按 PTX 地址空间分类（global/shared/local/const/param）实现语义正确。
- 支持 **可配置模块组合**：调度策略、并行模式、确定性模式、观测开关可由配置选择，不需要修改 SIMT 执行核心逻辑。

## 非目标（Non-goals）
- 不追求 cycle-accurate pipeline、scoreboard、带宽争用、cache 命中/替换。
- 不建模 NoC/VA/page table 的时序；如需地址翻译，仅作为可选的抽象检查层。
- 不承诺并行模式下严格的“事件全序与 bit-exact trace 一致性”（见确定性模式）。

---

## 顶层架构（Top-level Architecture）

### 核心组件
- **ComputeEngine**
  - 负责 kernel submit：构建 KernelContext、枚举 CTA work items，并提交到 GlobalScheduler。
  - 负责 kernel complete 判定：当所有 CTA 完成时 signal DependencyTracker。

- **GlobalScheduler（CTA Distributor）**
  - 线程安全队列/调度器：维护待执行 CTA 列表，供 SM workers 并发获取。

- **SMWorker（每 SM 一个软件线程）**
  - 线程循环：获取 CTA → 初始化 CTAContext → 在本 SM 上调度并推进 warps → 完成 CTA → 获取下一 CTA。
  - SM 内持有：WarpScheduler、Executor、Units（ExecCore/ControlUnit/MemUnit）与必要的 SM-local 状态。

- **MemoryModel（No-cache）**
  - 全局共享：global/const/param 由共享对象承载。
  - CTA 私有：shared 绑定到 CTAContext。
  - Thread 私有：local 绑定到 ThreadContext（或 warp/lane 私有）。

- **Observability（ObsControl）**
  - 并发安全事件入口：多个 SMWorker 并发 emit。
  - 支持全局单调时间戳（逻辑时钟）。

---

## 线程模型（Threading Model）

### 线程与职责
- **Runtime/主线程**：驱动 streams/queues/DependencyTracker；向 engines 提交 ready 命令。
- **ComputeEngine 驱动线程（可与主线程同一）**：处理 kernel submit 与完成判定。
- **SMWorker 线程池**：大小 = `sm_count`，每个线程模拟一个 SM。

> 实践建议：SMWorker 的生命周期与 Runtime 对齐（Runtime 初始化时启动，析构时停止），避免每次 kernel submit 创建/销毁线程带来的开销与不确定性。

### 共享状态与所有权
- **GlobalScheduler**：共享（多生产者/多消费者）。
- **GlobalMemory/ConstMemory/ParamMemory**：共享（多 worker 并发访问）。
- **CTAContext（含 shared、barrier 状态、warp 列表）**：由某个 SMWorker 独占直到 CTA 完成。
- **ThreadContext/warp regs/preds**：由 SMWorker 独占。

## 调度与执行流程（Scheduling & Execution Flow）

### 1) Kernel submit（ComputeEngine）
概念流程：
```text
submit_kernel(cmd):
  kernel_id = allocate
  kernel_ctx = build_kernel_context(module, entry, grid/block, params)
  for each cta_id in grid:
    cta_work_item = { kernel_id, cta_id, entry, launch_dims, params_handle }
    GlobalScheduler.submit(cta_work_item)
  mark kernel_ctx as running
```

### 2) CTA 分发（GlobalScheduler）
接口（概念）：
```text
submit(cta_work_item)
try_acquire(sm_id) -> optional<cta_work_item>
mark_done(cta_id)
```

基线策略（可替换）：
- FIFO：按 grid 枚举顺序出队。
- 可扩展：按资源需求/占用做更复杂策略，但不改变 contracts。

### 3) SMWorker 执行（每 SM 一个线程）
概念流程：
```text
worker_loop(sm_id):
  while running:
    item = GlobalScheduler.try_acquire(sm_id)
    if !item: wait_or_yield(); continue

    cta_ctx = init_cta_context(item)
    while !cta_ctx.done:
      warp_id = warp_scheduler.pick_ready_warp(cta_ctx)
      if !warp_id: break_or_blocked_handling()
      step = executor.step_warp(sm_state, cta_ctx, warp_id)
      handle_step_result(step)

    GlobalScheduler.mark_done(item.cta_id)
```

要点
- `executor.step_warp()` 的语义保持不变（详见 06 SIMT Core）。并行化不应把“并发逻辑”塞进 Executor。
- CTA 内 barrier/atom/fence 的语义由 MemoryModel + CTAContext（barrier 状态）保证。

### 4) Kernel 完成判定（ComputeEngine）
- ComputeEngine 维护 `kernel_remaining_cta_count`。
- 每个 CTA 完成后递减计数；当计数归零时 kernel complete。

## 内存与同步语义（No-cache, PTX AddrSpace-based）

### 地址空间（必须）
- `global`：跨 CTA/跨 SM 共享。
- `shared`：CTA 私有，barrier 后同 CTA 内可见。
- `local`：thread 私有。
- `const`：只读共享。
- `param`：只读（每次 launch 独立的 param blob/layout）。

### 并行下的实现要求
- **GlobalMemory**：必须并发安全。
  - 基线：全局互斥锁保护读写/atomic（简单但可能影响性能）。
  - 可选优化：按页/区间分段锁；atomic 走独立路径。

- **Atomic**：在 No-cache 模型下，原子语义可通过“对同一地址的序列化”实现。
  - 需求层面只要求功能正确与可复盘，不做时序精确。

- **Fence / Memory Model**：提供保守的顺序与可见性。
  - 基线：对 global 访问的 fence 通过全局锁/序列化点实现（例如 fence 触发一次全局同步/或发布-获取的保守实现）。
  - 强烈建议在文档/实现中明确：支持的 fence kind/scope，以及未实现部分的错误/降级策略。

- **Barrier**：CTA-level barrier 只在 CTAContext 内实现，不跨 CTA。

> 备注：cache 结构不仿真意味着“访存延迟/带宽争用”不建模；因此并行执行主要用于提高仿真吞吐与模拟多 SM 的程序间/CTA 间并发效果。

## 可观测性与时间戳（Observability & Timestamps）

### 并发 emit 的要求
- `ObsControl.emit(Event)` 必须线程安全。
- TraceBuffer 追加写必须支持并发（无锁 ring buffer 或加锁队列均可）。

### 时间戳（Event.ts）策略
提供两种模式（配置选择）：
1) **Parallel/Non-deterministic**：
   - `Event.ts = LogicalClock.fetch_add(1)`（全局原子单调递增）。
   - 只保证单调性，不保证与“仿真逻辑步 tick”的对应。

2) **Deterministic（用于 CI / golden traces）**：
   - 方案 A（简单）：禁用并行（`sm_count=1` 或 `parallel=false`），让 `tick` 驱动全序。
   - 方案 B（锁步并行）：每个 tick 内允许每个 SM 推进固定数量 step，然后 barrier 同步，tick++。

建议基线：优先实现方案 A，确保现有 golden trace 可复用；方案 B 作为后续扩展点。

## 配置项（Configuration）

建议新增（或在现有配置结构中扩展）以下键：
```text
sim.sm_count: int            // SM worker 数量
sim.parallel: bool           // 是否启用并行 worker
sim.deterministic: bool      // 是否启用确定性模式（基线：parallel=false）

sim.cta_scheduler: string    // FIFO | (future)
sim.warp_scheduler: string   // round_robin | (future)

obs.ts_mode: string          // logical_clock | tick
obs.thread_safe: bool        // 强制开启（并行时必须 true）
```

## 不变量与错误处理（Invariants / Error Handling）
- CTAContext 的 shared/barrier 状态不得跨 SMWorker 共享。
- 任何对 global/const/param 的写入必须经由 MemoryModel（禁止绕过）。
- 并行模式下，错误/诊断仍需定位到 `{kernel/cta/warp/thread, pc, source_location}`。
- 当任一 SMWorker 遇到不可恢复错误：
  - 基线策略：标记 kernel failed，并通知 Runtime/DependencyTracker 终止该 kernel 的后续推进。

## 迁移与实现步骤（Migration Plan）

### Step 1：抽取 CTA 枚举与提交
- 将“枚举 grid/block → CTA 列表”的逻辑从 SIMT 执行路径抽到 ComputeEngine。
- SIMT Core 保持 `step_warp()` 契约不变。

### Step 2：引入 GlobalScheduler + SMWorker（串行实现）
- 先实现 `sm_count=1` 的 worker 版本（仍单线程），确保行为与现有路径一致。

### Step 3：开启 N workers 并行
- 引入线程池与 stop token。
- 使 MemoryModel/ObsControl 并发安全。

### Step 4：确定性回归策略
- 基线：`deterministic=true` 时强制 `parallel=false`。
- 后续：可选 lock-step 并行确定性。

---

## 验收用例（Acceptance / Tests）
- `sm_count=1` 与旧实现结果一致（功能与 trace 主序列一致）。
- `sm_count>1` 时：
  - CTA 能分配到不同 SMWorker 并发完成（Trace 中体现不同 `sm_id`）。
  - 无跨 CTA 通信的 kernel：输出 buffer 与 `sm_count=1` 一致。
- 并行下无数据竞争：TSAN/或等价检查（如可用）不报错；至少通过压力测试（多 CTA + 大量 global 读写）。

---

## 风险与权衡（Risks / Trade-offs）
- 全局锁的 GlobalMemory 实现会降低并行吞吐；但能保证正确性与实现简洁，适合作为基线。
- 并行调度带来 trace 非确定性；需要确定性模式支撑 golden tests。
- fence/atomic 的语义边界必须明确，否则容易出现“跑得动但语义不可信”的情况。
