# 02 Frontend：Descriptor-driven 指令解码（必须：按 inst desc 配置解析 operands）

说明
- 本文实现需求：“对用户输入的 PTX 文本里边的指令解析，**必须**按照 PTX 指令描述配置（inst desc）进行”。
- 这里的“解析”特指：operand 的 `kind/type/addr` 形态判定与语法约束不再由 Parser 硬编码规则决定，而由 DescriptorRegistry 中的 `InstDesc.operand_kinds` 驱动。
- 目标是让“支持哪些指令形态/operand 组合”完全由 inst desc 数据集定义，并把错误尽可能前移到 decode 阶段（而不是执行期 lookup miss）。

本次补充澄清（与“模式 B”一致的实现选择）
- PTX op → IR op 的映射由**单独的 PTX ISA map 配置文件**决定。
- 读入 PTX 后经过一条独立的“PTX→IR 映射/解析”处理，直接产出 IR 形态的 `InstRecord`。
- 因此不强制引入/暴露 `InstRecordRaw` 作为 Contracts；token 化中间态可以是实现细节。

重要澄清：PTX op / IR op / uops 的分层
- 本项目中存在两类“op”：
  - PTX op：来自用户输入 PTX 文本的 `opcode`（例如 `mov/ld/add`），以及其变体（`.u32/.u64` 等）。
  - IR op：执行系统内部用于查表与展开的 canonical 操作名（建议使用“通用 IR op”集合，稳定且跨 PTX 版本复用）。
- uops（micro-ops）是 IR op 的展开结果。

为满足“用户只关心 PTX8 op → 通用 IR op，不需要关心 uops 组合”的使用方式，本设计采用**强制两层配置**：
- PTX ISA map：负责 PTX op 形态 → `ir_op`（也即原 inst desc 的 opcode key）映射
- inst desc（IR semantics）：负责 `ir_op` → uops[] 展开

---

## 02.0 配置分层（唯一必选模式）

将“PTX 版本差异”与“通用语义实现”解耦：

1) PTX ISA 映射配置（用户关注，必须提供/选择）
- 作用：把 PTX8 的指令形态映射到通用 IR op（并给出 operand_kinds，用于 descriptor-driven decode）。
- 示例结构（概念）：
```text
PtxIsaMapEntry {
  ptx_opcode: string
  type_mod: string
  operand_kinds: vector<string>
  ir_op: string
}
```

2) IR 语义库（框架/默认提供，沿用“原 inst desc”或内置）
- 作用：把 `ir_op` 展开成 uops[]（micro-op templates）。
- 示例结构（概念）：
```text
IrSemEntry {
  ir_op: string
  operand_kinds: vector<string>   // 可选：若同一 ir_op 存在多形态
  uops: vector<UopTemplate>
}
```

原则
- 用户选择 PTX8：只需要提供/选择一套 PTX ISA 映射数据集；无需写 uops。
- inst desc 不再承担“描述 PTX 文本长什么样”的职责，它只描述 IR op 的执行语义（uops 模板）。
- 框架提供稳定的通用 IR 语义库，确保跨 PTX 版本复用。

---

## 02.1 背景与问题

当前实现（代码）
- Parser 在 `parse_inst_line()` 内部直接调用 `parse_operand()`：根据 token 形态硬编码推断 `OperandKind`（`%r/%rd/%p/数字/[...]` 等）。
- DescriptorRegistry.lookup 依赖 `InstRecord.operands[].kind` 形成 signature（`operand_kinds`）后做匹配。

痛点
- “哪一种 token 应该当作 addr/symbol/imm/reg”由 Parser 决定，而不是由 descriptor 配置决定；导致：
  - descriptor 数据集无法作为“语法/形态约束”，只能作为执行期语义表。
  - 形态歧义（例如 `[name]` 到底是 symbol 还是 addr）无法由 descriptor 决策。
  - 未来扩展更多 operand form 时 Parser 会变成 hard-coded PTX 语法解释器。

---

## 02.2 目标（Goals）
- Parser 只做**词法级拆分**：`pred/opcode/modifiers/operand_tokens`，不推断 operand kind。
- 引入独立的 **PTX→IR 映射/解析（PtxIsaMapper）**：
  - 输入：token 化后的 PTX 指令行 + PTX ISA map registry。
  - PTX ISA map 决定：候选指令形态（operand_kinds）以及其映射到的 `ir_op`。
  - 输出：IR 形态的 `InstRecord`（其 opcode 语义为 IR op）+ typed operands；或 `Diagnostic`。
- 错误可定位：诊断必须携带 `SourceLocation`（至少行号）和可选 inst_index。
- 行为 deterministic：同一 PTX 文本 + 同一 descriptor 集合 → decode 结果唯一。

非目标（Non-goals）
- 不在本需求内完成：完整 PTX 语法覆盖（.pragma/.loc/复杂 expression）。
- 不强制要求 descriptor 具备完整的 modifier（rounding/sat/volatile 等）匹配；但设计需留好扩展点。

## 02.3 数据契约（Contracts）

### 02.3.1 Token 化中间态（实现细节，不强制作为 Contracts）
实现上需要一份“token 化指令行”的中间结构，字段通常包含：
```text
TokenizedPtxInst {
  ptx_opcode: string
  mods: { type_mod, space, flags... }
  pred: optional { pred_id, is_negated }
  operand_tokens: vector<string>
  dbg: optional { file, line, column }
}
```

说明
- `operand_tokens` 不做 kind 推断。
- token 保留必要原貌（如 `[%rd0+4]`），以便由 operand_kinds 驱动解析。
- 该结构不必进入公开 contracts；可以只存在于 Parser/Mapper 的实现层。

### 02.3.2 InstRecord（现有契约）
PTX→IR 映射/解析步骤的输出为 `InstRecord`，以复用现有 Instruction System 与执行系统。

新增约定（推荐）
- `InstRecord.opcode` 在进入执行系统后语义为 `ir_op`（canonical op）。
- PTX opcode/type_mod 可在诊断/观测中保留；如未来需要长期保真，可再扩展 `InstRecord` 字段。

## 02.4 模块与流程（Flow）

### 02.4.1 新流程（必选）
```text
PTX text
  -> Parser: tokenize -> tokenized inst stream
  -> Binder: bind_first_kernel (labels/params/layouts；可选)
  -> PtxIsaRegistry.load(ptx_isa_map)
  -> PtxIsaMapper: map+parse(tokenized, ptx_isa) -> KernelImage{insts: InstRecord(IR op)}
  -> IR Semantics: lookup(ir_op) + expand(uops)
  -> SIMT execute
```

说明
- PTX ISA map 的加载时机必须在 PtxIsaMapper 之前。
- Binder 当前实现较薄（仅选择 kernel），未来可继续承担 label/symbol 绑定；本设计不强制改变 Binder 的职责边界。

### 02.4.2 最小侵入实现路径（推荐的落地策略之一）
若希望最小侵入改造现有代码：
- Parser 仍可维持现有“产出 KernelImage 的骨架”的行为，但指令行不再类型化 operands。
- token 化结果通过 side-table 或内部结构传递给 mapper（避免扩展 `InstRecord` 暂存 tokens）。

## 02.5 InstDecoder 设计

> 术语更新：这里更准确的是 “PTX→IR 映射/解析（PtxIsaMapper）”。

### 02.5.1 API（概念）
```text
map_and_parse(inst: TokenizedPtxInst, ptx_isa: PtxIsaRegistry) -> Result<InstRecord, Diagnostic>

map_and_parse_kernel(insts: vector<TokenizedPtxInst>, ptx_isa) -> Result<vector<InstRecord>, Diagnostic>
```

### 02.5.2 依赖的 PTX ISA map registry 能力（新增概念）
Mapper 不能依赖执行期的 `lookup(InstRecord)`（因为此时还没有 typed operands，且还没得到 ir_op）。
需要 PTX ISA map 提供候选枚举 API（概念）：
```text
candidates(opcode: string, type_mod: string, operand_count: int) -> vector<const InstDesc*>
```

候选过滤最小维度
- `opcode` 必须匹配（canonicalized）。
- `type_mod`：优先 exact，再考虑 descriptor 的空串通配。
- `operand_count` 必须相等。
-（可选扩展）`mods.space/flags` 也可进入候选过滤与排序。

### 02.5.3 映射/解析算法（deterministic）
对每条 token 化 PTX 指令：
1) `opcode` canonicalization（例如 lower-case）。
2) 从 PTX ISA map 获取 candidates（按优先级排序：exact type_mod > wildcard）。
3) 对每个 candidate：
   - 如果 `operand_tokens.size != operand_kinds.size`：reject。
   - 逐 operand 尝试 `parse_operand_by_kind(kind, token, inst_raw.mods)`：
     - 任一 operand parse 失败 → reject。
  - 全部成功 → 记录为 match，并得到 candidate.ir_op。
4) 如果 match 数量为 1：生成 IR 形态 `InstRecord{ opcode=ir_op, operands=typed }` 并返回。
5) 如果为 0：返回 `Diagnostic{ code=DESC_NOT_FOUND | OPERAND_FORM_MISMATCH }`。
6) 如果 >1：返回 `Diagnostic{ code=DESC_AMBIGUOUS }`，并在 message 里列出候选 signature。

### 02.5.4 kind → token 解析规则（基线）
> 解析规则是“固定实现 + descriptor 选择 kind”，以达到“形态由配置决定”。未来如需让规则本身可配置，可扩展 schema。

建议基线规则（与当前实现对齐）：
- `reg`：`%r<N>` → `OperandKind::Reg + type=U32`；`%rd<N>` → `OperandKind::Reg + type=U64`。
- `pred`：`%p<N>` → `OperandKind::Pred`。
- `imm`：十进制或十六进制（允许前导 `-`），写入 `imm_i64`。
- `addr`：`[ %rd<N> ]` 或 `[ %rd<N> + <imm> ]` → `OperandKind::Addr`，base reg + offset imm。
  -（可选）允许 `[ %r<N> + imm ]`，但会强制扩展为 64-bit addr。
- `symbol`：非空 token（允许 `[name]` 形式，表示 param/global symbol）。

类型赋值规则（基线）
- 对于 `reg`：若 descriptor `type_mod` 是 `u64/s64`，允许将 `%rN` 的 `ValueType` 由 U32 提升为 U64（与当前 Parser 中的“u64 自动提升”一致），但必须 deterministic。
- 对于 `addr`：默认 `ValueType::U64`。

## 02.6 诊断与观测（Diagnostics + Observability）

### 02.6.1 Diagnostic codes（建议）
- `DESC_NOT_FOUND`：opcode/type_mod/operand_count 下无任何候选。
- `OPERAND_FORM_MISMATCH`：存在候选，但没有一个能把 tokens 按其 operand_kinds 全部解析成功。
- `DESC_AMBIGUOUS`：多个候选可成功解析（需要用户/descriptor 消歧）。
- `OPERAND_PARSE_FAIL`：token 语法非法（例如 `%r` 后非数字）。

### 02.6.2 Observability events（建议）
- category=`Fetch` 或 `Stream`：
  - `action=decode_begin/decode_end`（可带 inst_count）。
  - `action=decode_hit/decode_miss/decode_ambiguous`（extra 含 opcode/type_mod/operand_count）。

## 02.7 Schema 与数据集影响

现有 schema
- `operand_kinds` 仅是 string array（`reg/pred/imm/addr/symbol`）。

本需求下的最小要求
- descriptor 数据集需要覆盖“允许的 operand kind 组合”。
- 通过“候选尝试 + kind 驱动解析”实现 decode。

可选扩展（未来）
当需要表达更丰富的 operand 形态时，可扩展 descriptor：
- `operand_syntax`: 逐 operand 的更细粒度约束（例如 `addr{base_reg_width=64, allow_plus=true}`）。
- `mods`: 明确 rounding/sat/volatile/space 的匹配规则。

## 02.8 验收用例（Acceptance / Golden）

1) 形态由 descriptor 决定
- descriptor 只有 `ld.u64 (reg, symbol)`：
  - `ld.u64 %rd0, [param_name];` 应 decode 成 (reg, symbol)，而不是把 `[param_name]` 当 addr。

2) 歧义检测
- descriptor 同时存在：`mov.u32 (reg, imm)` 与 `mov.u32 (reg, symbol)`，且输入 `mov.u32 %r0, 0;` 只能匹配 imm，不应匹配 symbol。
- 输入 `mov.u32 %r0, foo;` 只能匹配 symbol。

3) 错误前移
- 输入出现未支持形态（例如 `add.u32 %r0, %r1, 1;` 但 desc 只支持 reg,reg,reg）：
  - decode 阶段直接报 `OPERAND_FORM_MISMATCH`，错误定位到行号。

4) Determinism
- 相同 PTX + 相同 desc 集合，多次 decode 产生相同 InstRecord 序列。
