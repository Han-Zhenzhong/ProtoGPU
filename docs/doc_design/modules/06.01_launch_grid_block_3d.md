# 06.01 Kernel Launch（3D grid/block + builtins）

说明
- 本文补齐 06 SIMT Core 在“kernel launch 维度建模与 builtin 提供”上的抽象逻辑。
- 设计基准：与 [doc_design/modules/07_runtime_streaming.md](07_runtime_streaming.md) 的 `KernelPayload{grid_dim, block_dim}` 对齐；若与 [doc_design/sequence.diagram.puml](../sequence.diagram.puml) 冲突，以图为准。

---

## 目标（Goals）
- 支持 3D `gridDim` / `blockDim`（`x/y/z`），并按 SIMT 语义把它们映射为 CTA/Warp/Thread 上下文。
- 支持 PTX builtin（至少）：`tid/ntid/ctaid/nctaid`，可扩展 `laneid/warpid/smid/gridid`。
- 不要求周期精确：这里定义的是功能正确与可复盘的抽象语义。

非目标
- 不在本阶段实现 occupancy/寄存器分配约束/真实硬件调度细节。

---

## 数据结构（Data Structures）

### dim3
```text
Dim3 { x: u32, y: u32, z: u32 }
```

约束（基线）
- 所有维度必须 >= 1。
- `block_dim.x * block_dim.y * block_dim.z` 为 block 内线程数。

### LaunchConfig（一次 kernel launch 的执行域）
```text
LaunchConfig {
  grid_dim:  Dim3
  block_dim: Dim3
  warp_size: u32           // 来自 sim config
}
```

### Builtins（每 thread/lane 可见的 builtin 值）
```text
Builtins {
  tid:    Dim3
  ntid:   Dim3
  ctaid:  Dim3
  nctaid: Dim3
  laneid: u32
  warpid: u32              // CTA 内 warp 序号
}
```

---

## 映射规则（Mapping Rules）

### 1) CTA（block）枚举
对每个 CTA：
- `ctaid.x ∈ [0, grid_dim.x)`
- `ctaid.y ∈ [0, grid_dim.y)`
- `ctaid.z ∈ [0, grid_dim.z)`

线性 CTA id（实现可选）
```text
cta_linear = (ctaid.z * grid_dim.y + ctaid.y) * grid_dim.x + ctaid.x
```

### 2) threadIdx（tid）枚举
对每个 thread：
- `tid.x ∈ [0, block_dim.x)`
- `tid.y ∈ [0, block_dim.y)`
- `tid.z ∈ [0, block_dim.z)`

线性 thread id（CTA 内）
```text
thread_linear = (tid.z * block_dim.y + tid.y) * block_dim.x + tid.x
```

### 3) Warp/Lane 划分
```text
laneid = thread_linear % warp_size
warpid = thread_linear / warp_size
warp_count = ceil(threads_per_block / warp_size)
```

约束
- 最后一个 warp 可能是部分填充：`active_mask` 只包含有效 thread 对应的 lanes。

## builtin 的提供方式（How to expose builtins）

需求：PTX 指令流需要能读取 builtin（如 `%tid.x`）。

设计选项（推荐按演进实现，保持可扩展）：

### 选项 A（推荐基线）：Special 作为 OperandKind
扩展 contracts：
```text
OperandKind += Special
Operand.special = { name: "tid.x" | "ctaid.y" | ... }
```

Frontend 在 tokenization/operand 解析时把 `%tid.x` 等识别为 `Special`。

语义：
- `mov.u32 %r0, %tid.x;` 通过 inst desc 展开为一个 `MOV` uop，输入为 `Special(tid.x)`。
- ExecCore 在读操作数时从 `Builtins` 读取对应值（按 lane 生成 lane-wise 值）。

优点
- 不需要在指令列表前插入“隐式写寄存器”的 prolog。
- trace 复盘更直接（能看到读取的是哪个 builtin）。

### 选项 B：Kernel prolog 物化内建寄存器
在每个 CTA 的每个 warp 启动前执行一次 prolog，把 builtin 写入约定寄存器（例如 `%r_tid_x`）。

优点
- contracts 无需扩展。

缺点
- 需要引入“约定寄存器号/命名”，对 frontend/映射更侵入。

## 与 SIMT Core 的接口（Integration Points）

在 [doc_design/modules/06_simt_core.md](06_simt_core.md) 的上下文模型中：
- CTAContext 持有 `ctaid/nctaid/block_dim`。
- WarpContext/ThreadContext 持有（或可计算）`tid/ntid/laneid/warpid`。

Executor 的 lane-wise 操作数读取需要可访问 Builtins：
```text
read_operand(op, lane) -> scalar
  if op.kind == Special: return builtins[lane][op.name]
  else: existing reg/imm/addr/symbol logic
```

---

## 错误处理与诊断（Diagnostics）
- 维度为 0：在 launch 时直接报 `runtime:E_LAUNCH_DIM`。
- `threads_per_block == 0` 或乘法溢出：报 `runtime:E_LAUNCH_OVERFLOW`。
- 若引入 builtin `Special`：未知 builtin 名称报 `units.exec:E_SPECIAL_UNKNOWN`，并携带 `inst_index`。

---

## 验收用例（Smoke tests / Golden traces）
- `block_dim=(32,1,1)`：单 warp 满填充，`laneid==tid.x`。
- `block_dim=(40,1,1)`：warp0 满，warp1 部分；warp1 的 `active_mask` 只包含 8 个 lane。
- 3D：`block_dim=(4,3,2)`，验证 `thread_linear` 与 `tid.{x,y,z}` 互相一致。
- builtin：kernel 读 `%ctaid.x/%tid.x` 写入 global，host 回读验证。
