# 07.01 Stream 输入（WorkloadSpec：数据输入/参数输入/launch/依赖）

说明
- 本文定义“用户如何输入 stream 命令序列”的抽象：用一个可重放的 workload 配置描述 COPY / KERNEL / EVENT。
- 该设计与 [doc_design/modules/07_runtime_streaming.md](07_runtime_streaming.md) 的命令模型一致，并复用 [doc_design/modules/09_kernel_io_and_abi.md](09_kernel_io_and_abi.md) 的 buffer/params 语义。

---

## 目标（Goals）
- 支持用户以结构化输入描述：
  - 数据输入（H2D / MEMSET）
  - 参数输入（KernelArgs）
  - kernel launch（含 3D grid/block）
  - stream 输入（多 stream、event record/wait、同步点）
- 支持最小回归：同一份 workload 在不同机器上可复现同样的 command 因果链、diag 与 trace。

非目标
- 不在此定义真实 CUDA API 细节；只给出与本项目 Runtime/Streaming 契约一致的“最小可实现形态”。

---

## WorkloadSpec（建议新增的用户输入文件）

建议 CLI 新增：`--workload <json>`（替代目前“仅跑单 kernel”的模式），其 schema 可以逐步演进。

概念结构
```text
WorkloadSpec {
  buffers: {
    host:   map<string, HostBufferDecl>
    device: map<string, DeviceBufferDecl>
  }
  modules: map<string, ModuleDecl>         // PTX 文件集合
  streams: map<string, StreamDecl>
}

HostBufferDecl { bytes: u64, init?: oneof { zeros, file, hex } }
DeviceBufferDecl { bytes: u64, align?: u64 }

ModuleDecl { ptx: string, ptx_isa: string, inst_desc: string }

StreamDecl {
  commands: CommandSpec[]                  // FIFO
}
```

命令结构（与 07 模块保持一致）
```text
CommandSpec = oneof {
  copy:   CopySpec
  kernel: KernelLaunchSpec
  event_record: { event: string }
  event_wait:   { event: string }
  sync:  { }                               // stream sync
}

CopySpec { kind: H2D|D2H|D2D|MEMSET, dst, src, bytes }
KernelLaunchSpec {
  module: string
  entry:  string
  grid_dim:  Dim3
  block_dim: Dim3
  args: map<string, ArgValue>
}

ArgValue = oneof { u32, u64, ptr_device("dev_buf_name"), bytes_hex }
```

注
- `ModuleDecl` 将 `ptx + ptx_isa + inst_desc` 绑定在一起，保证 workload 可重放（不会依赖默认资源路径）。
- `sync` 是一个显式同步点，便于脚本式回归与“取回结果后断言”。

## 参数输入（KernelArgs 生成）
- 具体 ABI/布局来源见 [doc_design/modules/09_kernel_io_and_abi.md](09_kernel_io_and_abi.md)。
- WorkloadSpec 的 `args` 只表达“按参数名赋值”。Runtime 负责：
  - 从 Frontend/Binder 得到 `ParamDesc layout`
  - 将 `args[name]` 按 layout 写入 blob（小端序）
  - `ptr_device` 会被解析成 `DevicePtr`（来自 device buffer 分配）

---

## 数据输入（Buffers + COPY）
- Host buffer 与 device buffer 的声明用于：
  - 提前分配并稳定命名（方便在 args/copy 中引用）
  - 允许把测试输入从二进制/hex 文件加载到 host buffer
- COPY 语义、越界/对齐/未分配诊断见 [doc_design/modules/09_kernel_io_and_abi.md](09_kernel_io_and_abi.md) 与 [doc_design/modules/04_memory.md](04_memory.md)。

---

## stream 输入（多 stream + 事件依赖）
- 每个 stream 有独立 FIFO。
- `event_record` / `event_wait` 的 ready 判定与可观测点见 07 模块。
- WorkloadSpec 的 `event` 用 string 命名，Runtime 内部分配 `EventId` 并建立映射。

建议的可观测事件（基线）
- `STREAM:cmd_enq/cmd_ready/cmd_submit/cmd_complete`
- `STREAM:event_record/event_wait_done`

---

## 与 Engines/SIMT 的边界（Boundary）
- Runtime/Streaming 只负责：
  - 解析 WorkloadSpec
  - 创建 buffers/streams/events
  - 将 commands 入队并调度 ready
- ComputeEngine/SIMT 负责：
  - KERNEL command 的执行推进
  - 维度与 builtin 规则见 [doc_design/modules/06.01_launch_grid_block_3d.md](06.01_launch_grid_block_3d.md)

---

## 错误处理（Diagnostics）
- WorkloadSpec schema 错误：`runtime:E_WORKLOAD_SCHEMA`
- 引用不存在的 buffer/module/event：`runtime:E_WORKLOAD_REF`
- launch 维度非法：`runtime:E_LAUNCH_DIM`
- COPY 越界/未分配：`memory:E_*`（并关联到 `cmd_id/stream_id`）

---

## 验收用例（Smoke tests）
- 单 stream：H2D → kernel(grid=1, block=32) → D2H → sync → assert
- 双 stream：stream0 recordEvent(E)；stream1 waitEvent(E) 后再 launch kernel
- 3D：grid=(2,2,1)、block=(16,2,1)，kernel 把 `ctaid/tid` 写回 global 供 host 校验
