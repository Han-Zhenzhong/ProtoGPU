# 04 SIMT：线程分歧与合流（Divergence + Reconvergence，里程碑 M5）

本文在 [docs/doc_design/ptx6.4_sm70/03_simt_predication_and_controlflow.md](03_simt_predication_and_controlflow.md) 的 M3/M4 基线（predication + uniform-only）之上，定义“支持 warp 内线程分歧执行”的 SIMT 设计：

- 对 predicated `bra` 支持真正的 divergence（taken/fallthrough 两条路径同时存在）。
- 通过 SIMT stack 实现路径分割与合流（reconvergence）。
- 明确与现有 contracts（PC=InstList index、LaneMask、StepResult.next_pc、Event.extra_json）的对齐方式。

相关模块设计：
- SIMT 核心抽象：[docs/doc_design/modules/06_simt_core.md](../modules/06_simt_core.md)
- Units 抽象（含 reconv 栈概念）：[docs/doc_design/modules/05_units.md](../modules/05_units.md)
- 规格基线（当前明确“不保证” reconvergence）：[docs/doc_spec/sm70_profile.md](../../doc_spec/sm70_profile.md)

---

## 1. 目标 / 非目标

目标（M5）
- 支持 `bra` 的 divergence：同一 warp 内 active lanes 对分支谓词可不一致。
- 支持合流（reconvergence）：分歧路径在控制流图的 post-dominator 处重新合并。
- 与现有执行主循环保持一致：`FETCH → lookup → expand → pred inject → dispatch → COMMIT`（commit 仍是 inst 级）。
- 可观测：trace 能复盘 mask/PC 的变化与 stack 行为。

非目标（本设计明确不做）
- 不引入 per-lane PC（仍坚持“warp 单 PC + active_mask + stack”模型）。
- 不承诺完整 PTX 控制流覆盖（call/ret with callstack、indirect branch、switch、异常）。
- 不在 M5 同时解决 barrier/原子与分歧交互的所有角落（仅保证与现有 barrier=尚未实现的规划不冲突）。

---

## 2. 术语与统一口径

- **PC**：`InstList` index（`using PC = std::int64_t`）。
- **active_mask**：当前正在执行“当前路径”的 lanes。
- **exec_mask**：某条 micro-op 实际生效的 lanes：`exec_mask = active_mask & uop.guard`。
- **Divergence**：对同一控制 op（`bra`），在 `active_mask` 内存在 taken 与 fallthrough 都非空的 lane 子集。
- **Reconvergence**：分歧后的多条路径在同一个 reconv 点重新合并，恢复为一个 `active_mask`。

关键约束（保持与现有 contracts 一致）
- 单条 inst 的 uops 视为不可拆分序列；所有 `next_pc` 的效果在 commit 阶段生效。
- Units 只执行 uops，不做分歧判定；分歧/合流编排归属 SIMT executor。

---

## 3. 设计总览

M4（现状）在 SIMT 层对 `bra/ret` 做 uniform-only 检测，遇到 `taken_mask != active_mask` 直接 `E_DIVERGENCE_UNSUPPORTED` fail-fast。

M5（本设计）将该 fail-fast 替换为：

1) **静态控制流分析**（kernel-level）：为每个可能产生 divergence 的 `bra` 预计算 reconv_pc（分支点的 immediate post-dominator）。
2) **SIMT stack 执行**：在执行期遇到 divergence 时把 warp 拆成两个“路径帧”，以 LIFO 顺序执行并在 reconv_pc 合并。
3) **lane-wise 退出语义**：`ret` 对 `exec_mask` 中的 lanes 生效（lane 退出），允许不同 lanes 在不同时间退出。

---

## 4. 控制流分析：为 `bra` 预计算 reconv_pc

### 4.1 为什么需要静态 reconv_pc

只靠运行时 “pc+1 / target” 无法稳定推导合流点：
- if-else、嵌套分支、循环都会导致 `pc+1` 不是 join。
- PTX 代码往往通过无条件 `bra` 跳转到 join label。

因此 M5 必须在执行前得到：

- `reconv_pc[pc] = ipdom(pc)`（pc 为 `bra` 的 InstList index）。

### 4.2 CFG 构建（KernelImage.insts → 图）

节点
- 每条 inst 对应一个节点 `N(pc)`。
- 追加一个 `EXIT` 节点（synthetic）。

边（successors）
- 普通指令：`pc -> pc+1`。
- `ret`：`pc -> EXIT`。
- `bra target`：
  - unconditional：`pc -> target`。
  - predicated（可能 divergence）：`pc -> target` 与 `pc -> pc+1`。

注意：本项目当前 `bra` 是否“predicated”可由 `InstRecord.pred`（来自 Parser）判断；SIMT 执行时真正的 taken/fallthrough 由 predication 注入后的 `uop.guard` 决定。

### 4.3 Post-dominator / ipdom 计算

对 CFG（含 EXIT）计算 post-dominator 集，进而求 immediate post-dominator（ipdom）。

输出
- 对每条 `bra` 指令 index 产出 `reconv_pc`：
  - 若 `ipdom(pc) == EXIT`，可将 `reconv_pc` 记录为一个专用 sentinel（例如 `kExitPc = kernel.insts.size()`），并由 SIMT 执行器在“到达 reconv_pc”时识别为“路径终结/warp 完成候选”。

失败策略
- 若 CFG 分析无法为某条 predicated `bra` 给出 ipdom：诊断 fail-fast（建议 `simt:E_RECONV_MISS`），以避免静默错误执行。

## 5. SIMT stack：路径分割与合流

### 5.1 数据结构（建议）

在 warp 上下文引入 SIMT stack：

```text
SimtFrame {
  pc: PC
  active: LaneMask
  reconv_pc: PC   // 本帧的合流点（ipdom）
}

WarpState (增量) {
  // 现有字段：pc/done/active/regs/preds...
  exited: LaneMask         // lanes 已执行 ret/exit（lane-wise）
  simt_stack: vector<SimtFrame>
}
```

约束
- 执行时“当前 warp 的 pc/active”来自 `simt_stack.back()`（top frame）。
- `warp.exited` 与 `frame.active` 必须一致维护：`frame.active &= ~warp.exited`。

### 5.2 初始化

对每个 warp：
- 计算 partial warp 的初始 mask：`active0`（现有实现已有）。
- `warp.exited = 0`。
- `simt_stack = [{ pc=0, active=active0, reconv_pc=EXIT }]`。

### 5.3 每步执行（step_warp）骨架

在进入 fetch 前，对 top frame 做规范化：

1) `frame.active &= ~warp.exited`。
2) 若 `frame.active` 为空：pop frame（该路径没有活跃 lane）。
3) 若 `frame.pc == frame.reconv_pc`：
   - pop frame（该路径到达 reconv 点，等待合并）。
   - 将其 active 合并到新的 top（若新 top 的 `pc` 也等于该 reconv_pc）；否则将合并后的 frame 重新压回（见 5.5）。
4) 若 stack 为空：warp 完成。

然后再进入现有的 `FETCH → lookup → expand → pred inject → dispatch → COMMIT`。

### 5.4 `bra` 的 divergence 处理（在 SIMT 层）

当 dispatch 到 control micro-op `BRA` 时，在 SIMT 层（不是 ControlUnit）计算：

- `taken = frame.active & uop.guard`
- `fallthrough = frame.active & ~uop.guard`

情况 A：`taken` 为空
- 该 `bra` 对本帧无效果（等价 no-op），commit 默认走 `pc+1`。

情况 B：`fallthrough` 为空
- uniform taken：commit 设置 `pc = target_pc`（行为与 M4 一致）。

情况 C：二者均非空（divergence）
- 取 `reconv_pc = precomputed_reconv_pc[pc]`（由 §4 得到）。
- 将当前 top frame 原位改写为 **fallthrough 帧**：
  - `frame.pc = pc+1`
  - `frame.active = fallthrough`
  - `frame.reconv_pc = reconv_pc`
- push 一个 **taken 帧** 到 stack 顶：
  - `{ pc = target_pc, active = taken, reconv_pc = reconv_pc }`

接下来执行哪个路径：
- 本设计选择 **taken-first**（LIFO，执行 top），因此下一次 fetch 将从 `target_pc` 以 `taken` mask 开始。

可观测建议
- 记录一次 `EventCategory::Ctrl`，action 建议为 `SIMT_SPLIT`，并在 `extra_json` 写入：
  - `pc`, `target_pc`, `reconv_pc`, `mask_active`, `mask_taken`, `mask_fallthrough`, `stack_depth_after`。

### 5.5 合流规则（merge）

当一个 frame 达到 `frame.pc == frame.reconv_pc` 时：

- pop 该 frame，并将它的 `active` 合并到“同 pc 的等待帧”。

最小规则（足以覆盖 if-else / 嵌套 if）
- 若 pop 后新的 top frame 满足 `top.pc == popped.pc`：
  - `top.active |= popped.active`
  - `top.reconv_pc` 取二者更外层的 reconv（通常应相同或由 CFG 保证嵌套正确）。
- 否则：
  - 说明 pop 出来的 frame 抵达 reconv 点，但其他路径还未抵达该点；需要把它作为“等待帧”保留。
  - 具体做法：把 popped frame 作为新的 top 再 push 回去（并立刻 pop 会造成死循环，因此必须在下一轮与其他帧交错执行）。

实现提示（为了避免复杂性）
- 推荐引入一个显式 `SimtFrameKind`：`Path` vs `Join`，让 join frame 只用于合并与等待，不参与普通指令执行；当 join 变为 top 且合并条件满足时才“激活并开始执行”。
- 若暂不引入 kind，则必须在 step 前的“规范化阶段”里处理好 `pc==reconv_pc` 的 pop/merge，保证不会在 reconv 点重复执行错误的路径切换。

## 6. `ret` / lane-wise 退出语义

M4 中 `ret` 等价于 warp 退出（`warp.done=true`）。在支持 divergence 后，这会产生错误：一个 lane 在某条分歧路径上返回不应强制其他 lanes 立即退出。

M5 冻结语义
- `RET` 对 `exec_mask` 生效：
  - `warp.exited |= exec_mask`
  - 当前 frame 的 `active &= ~exec_mask`
- 当某个 frame 的 `active` 变为空：该路径结束，pop frame。
- warp 完成判定：
  - `warp.exited` 覆盖了初始 active0（或 `warp.exited == lane_mask_all(warp_size)` 对满 warp 情况）
  - 且 `simt_stack` 为空。

诊断
- 对 `RET` 的 operand/arity 校验仍由 ControlUnit 负责（与 M4 一致）。

## 7. 与现有模块的接口边界

### 7.1 SIMT vs ControlUnit 的职责边界

保持一致的边界（推荐）
- ControlUnit：只负责“单条 control uop 的局部语义”（如解析 target、返回 next_pc、校验 operand）。
- SimtExecutor：负责 divergence 检测、stack 更新、合流编排。

这与当前实现一致点
- `next_pc` 提交点仍在 SIMT（inst 级 commit）。

### 7.2 WarpState 扩展的最小化

现状 `WarpState` 在 [include/gpusim/units.h](../../include/gpusim/units.h) 中定义。M5 需要增加：
- `LaneMask exited`
- `std::vector<SimtFrame> simt_stack`

并将 `warp.done` 从“写死 bool”转为“由 exited+stack 计算”，或保持 bool 但只作为缓存字段（以免并发路径下状态不一致）。

## 8. 观测（Trace）与诊断

### 8.1 Trace 事件建议

现有事件结构允许通过 `extra_json` 附带结构化信息（不破坏 schema）：

- `SIMT_SPLIT`：发生 divergence（见 §5.4）。
- `SIMT_POP`：路径帧被 pop（active 空或到达 reconv）。
- `SIMT_MERGE`：在 reconv 点合并 masks。

事件字段建议
- `pc`：分歧点/合流点 pc。
- `lane_mask`：对应该事件涉及的 mask（例如 taken/fallthrough/merged，可放 extra_json）。

### 8.2 诊断码建议（新增）

- `simt:E_RECONV_MISS`：某条 predicated `bra` 发生 divergence，但缺少 `reconv_pc[pc]`。
- `simt:E_RECONV_INVALID`：reconv_pc 不在合法范围（且不是 EXIT sentinel）。

保留
- `simt:E_NEXT_PC_CONFLICT`：同一 inst 多个 control uop 提供冲突 next_pc（仍需防御）。

## 9. 验收用例（建议新增 targeted tests）

建议在现有 `tests/unit/simt_predication_controlflow_tests.cpp` 风格基础上新增：

1) 最小 if-else divergence + join
- warp_size=2
- lane0 predicate=true, lane1=false
- 结构：`@p bra L_then; ... else ...; bra L_join; L_then: ...; L_join: ret;`
- 预期：完成，且两 lane 在 join 后 `active_mask` 合并。

2) 嵌套分歧
- 两层 predicate 造成嵌套 split，验证 stack 深度与 merge 顺序。

3) Divergent ret（lane-wise）
- 只有部分 lanes 执行 `ret`，其余 lanes 继续执行后续指令并最终退出。

4) 失败路径
- 人工构造 CFG 分析拿不到 ipdom 的形态（若存在），触发 `E_RECONV_MISS`。

## 10. 分阶段落地建议

- M5.1：CFG + ipdom 预计算（不改执行器），以 unit test 验证 reconv_pc map。
- M5.2：SIMT stack（先支持 `bra` divergence + join，ret 仍 warp-wide）用于 bring-up。
- M5.3：lane-wise `ret` 与 warp 完成判定修正。
- M5.4：补齐 trace 事件与稳定的回归测试（golden trace 可选）。
